## Open

### P1 — Remote Terraform state backend
- **What:** Both stacks hardcode `backend "local" {}` (`terraform/infra/backend.tf`,
  `terraform/bootstrap/backend.tf`). `-backend-config` cannot change the backend
  *type*, so CI on ephemeral runners has no cross-run state persistence — a second
  Apply would try to recreate everything.
- **Why deferred:** Needs the team's backend choice (S3+DynamoDB / azurerm / GCS /
  Terraform Cloud) and the bucket/account details.
- **Follow-on:** Once remote, replace the Makefile's `terraform output -json …`
  env-var shuttle (`kubeconfigs`, `namespace_config`) with `terraform_remote_state`.
- **Size:** M.

### P1 — Rotate and externalize credentials
- **What:** `infrastructure/base/ako/ako.yml` commits a real AVI
  username/password/CA (base64) and ships it to every cluster as
  `cluster-avi-secret`. The supervisor-side secret is already Terraform-fed
  (`charts/bootstrap-tenant` → `istio-ako-avi-secret`); the workload-side secret
  needs the same treatment (or an external-secrets pattern).
- **Action:** Rotate the leaked credentials (they are in git history), then wire the
  workload secret through Terraform/external-secrets instead of git.
- **Owner note:** Intentionally left untouched on request.
- **Size:** M.

### P2 — Lock down tenant AppProjects
- **What:** The generated `type: tenant` AppProject grants `sourceRepos: '*'`, all
  destinations, and a `clusterResourceWhitelist: '*'/'*'` — not actually locked
  down (README previously over-claimed this; now corrected).
- **Action:** Scope `sourceRepos` to the known repo URL(s), drop the
  cluster-resource grant for tenants, set an explicit `namespaceResourceWhitelist`.
- **Blocker:** Scoping `sourceRepos` needs the repo URL available in the infra run
  (currently only a bootstrap var). Either pass it to infra or template from
  `argocd/repo-config.yaml`.
- **Files:** `terraform/infra/templates/appproject.yaml.tftpl`, `generate.tf`.
- **Size:** M.

### P2 — Make `istio-ako-patch` reusable (apps-tree variable injection)
- **What:** `apps/base/istio-ako-patch/overlay.yml` hardcodes the PackageInstall
  name `dev1-cluster-ako`, so the base is single-cluster.
- **Why deferred:** The apps Kustomize tree has no variable injection — fixing this
  means adding a `vars` ConfigMap + a `cluster-var-injector`-style component on the
  apps side (the infrastructure tree already has this).
- **Consideration:** This resource targets `vmware-system-tkg`; evaluate whether it
  belongs in the infrastructure tree (which already has injection) rather than apps.
- **Size:** M.

### P2 — CI: add Terraform checks to `validate.yml`
- **What:** `.github/workflows/validate.yml` currently only runs `kustomize build`
  + the cluster-details path check.
- **Action:** Add `terraform fmt -check -recursive terraform/` and
  `terraform validate` (init `-backend=false`) for `infra` and `bootstrap`.
- **Size:** S.

### P3 — `apply.yml` should stage deletions of generated files
- **What:** The "Commit generated files" step uses `git add <paths>`, which does not
  stage deletions. If a tenant is removed from `tenants.yaml`, its stale
  `argocd/projects/<name>.yaml` (and `clusters/<name>/vars/`) linger.
- **Action:** Use `git add -A -- <paths>` for the generated dirs.
- **Size:** S.

### P3 — `infrastructure/clusters/infra-1/vars/`
- **What:** `kustomization.yaml` is committed but its `tenant-vars.yaml` is not
  (it is regenerated by `make apply-infra`). Harmless today — nothing builds the
  infra tenant's vars dir — but untidy.
- **Action:** Either commit the rendered `tenant-vars.yaml` after the next apply, or
  skip rendering vars for the infra tenant if it will never host workload clusters.
- **Size:** S.

### P3 — Add a `prod` profile when the first prod cluster lands
- **What:** Only `profiles/dev` (infra + apps) exists. The `infra-1` tenant is
  `environment: prod` but has no cluster dir yet.
- **Action:** Copy `profiles/dev` → `profiles/prod` and add the real prod values in
  `infrastructure/components/envs/prod` when needed.
- **Size:** S.

### P3 — Document the `vars/` directory reservation
- **What:** `infrastructure/clusters/{project}/vars/` holds the Terraform-rendered
  tenant-vars and is a sibling of the `{namespace_ref}` directories. A namespace
  literally named `vars` would collide.
- **Action:** One-line note in the docs / template README.
- **Size:** XS.
